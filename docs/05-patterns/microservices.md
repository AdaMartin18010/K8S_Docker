# 微服务模式

> 基于容器的微服务架构设计

---

## 微服务特征

| 特征 | 说明 |
|------|------|
| **单一职责** | 每个服务只负责一个业务功能 |
| **独立部署** | 服务可以独立部署和扩展 |
| **去中心化** | 每个服务有自己的数据存储 |
| **容错性** | 服务故障不影响整体系统 |
| **自动化** | CI/CD 自动化部署 |

## 设计原则

### 1. 服务拆分

```
# 按业务能力拆分
┌─────────────────────────────────────┐
│           Monolith                  │
│  (User + Order + Payment + Inventory)│
└─────────────────┬───────────────────┘
                  │ Split
                  ▼
┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
│  User   │  │  Order  │  │ Payment │  │Inventory│
│ Service │  │ Service │  │ Service │  │ Service │
└─────────┘  └─────────┘  └─────────┘  └─────────┘
```

### 2. 数据管理

- 每个服务有自己的数据库
- 避免直接访问其他服务的数据库
- 通过 API 进行数据交互

### 3. 服务发现

```yaml
# 使用 K8s Service
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service
  ports:
    - port: 80
      targetPort: 8080
```

### 4. 配置管理

```yaml
# ConfigMap 配置
apiVersion: v1
kind: ConfigMap
metadata:
  name: user-service-config
data:
  DATABASE_URL: "postgres://user-service-db:5432/users"
  CACHE_TTL: "300"
```

## 部署模式

| 模式 | 说明 | 适用 |
|------|------|------|
| **单 Pod 多容器** | 相关服务共享 Pod | 紧密耦合的服务 |
| **单 Pod 单容器** | 每个服务独立 Pod | 标准微服务 |
| **多副本** | 水平扩展 | 高流量服务 |

---

## 最佳实践

1. **健康检查**: 每个服务都要有 liveness/readiness 探针
2. **资源限制**: 设置合理的 requests/limits
3. **日志规范**: 结构化日志，统一格式
4. **监控指标**: 暴露 Prometheus 指标
5. **熔断降级**: 使用 Service Mesh 或客户端库
